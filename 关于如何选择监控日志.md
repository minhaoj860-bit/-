## 1. rsyslog（系统自带）

### 原理
```
服务器 rsyslogd → TCP/UDP 514端口 → 本地 rsyslogd → 应用解析
```

### 配置方法
**服务器端：**
```bash
# /etc/rsyslog.d/40-nginx-error.conf
action(type="omfwd" target="10.0.1.3" port="514" protocol="tcp")

# 重启服务
systemctl restart rsyslog
```

**本地端：**
```bash
# 安装 rsyslog
apt install rsyslog

# /etc/rsyslog.conf 启用接收
module(load="imtcp")
input(type="imtcp" port="514")
```

### 优点
- 系统自带，无需安装
- 内核级别，性能极高
- 支持 TCP（可靠）/ UDP（高速）
- 有缓冲机制，断线自动重试

### 缺点
- 配置语法复杂
- 调试困难
- 单点故障，无 ACK 确认
- 消息格式固定

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| 发送成功但收不到 | 防火墙阻止 | `iptables -I OUTPUT -p tcp --dport 514 -j ACCEPT` |
| rsyslog 无法启动 | 配置文件语法错误 | `rsyslogd -f /etc/rsyslog.conf -N1` 检查语法 |
| 日志乱序丢失 | UDP 模式问题 | 改用 TCP 协议 |

## 2. Filebeat（⭐推荐）

### 原理
```
服务器: Filebeat 监控日志文件 → TCP 端口 → 本地服务
         ↓
       本地磁盘队列（不丢日志）
```

### 配置方法
**安装（服务器端）：**
```bash
curl -L -O https://artifacts.elastic.co/downloads/beats/filebeat/filebeat-8.x-amd64.deb
dpkg -i filebeat-*-amd64.deb
```

**配置 /etc/filebeat/filebeat.yml：**
```yaml
filebeat.inputs:
- type: log
  enabled: true
  paths:
    - /var/log/nginx/error.log
  fields:
    service: nginx
    server: 101.133.135.3

output.tcp:
  hosts: ["10.0.1.3:9000"]

processors:
- decode_json_fields:
    fields: ["message"]

filebeat.inputs:
- type: log
  close_inactive: 5m
  ignore_older: 12h
  scan_frequency: 10s
```

**本地端（Node.js）：**
```javascript
const net = require('net');
const server = net.createServer((socket) => {
  socket.on('data', (data) => {
    const lines = data.toString().split('\n');
    lines.forEach(line => {
      if (line.trim()) parseLog(line);
    });
  });
});
server.listen(9000);
```

### 优点
- 有重试机制，断线自动重连
- 有磁盘队列，服务重启不丢日志
- 资源占用极低（~5% CPU）
- 支持 SSL/TLS 加密
- 自动追踪文件（支持日志轮转）

### 缺点
- 需要额外安装
- 配置有学习成本
- 默认只能输出到 ES/Logstash，需自定义 output

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| Filebeat 启动失败 | YAML 格式错误 | `filebeat test config -c /etc/filebeat/filebeat.yml` |
| 日志重复发送 | registry 文件损坏 | `rm -f /var/lib/filebeat/registry/registry && systemctl restart filebeat` |
| 连接被拒绝 | 目标端口未开放 | 本地先 `nc -l 9000` 测试 |

## 3. Fluentd

### 原理
```
服务器: inotify → Fluentd → TCP 24224 → 本地 Fluentd → 存储
         ↓
       文件/内存缓冲
```

### 配置方法
**安装：**
```bash
curl -fsSL https://td-agent-package-browser.herokuapp.com/ubuntu/jammy/install.script.sh | bash
```

**配置 /etc/td-agent/td-agent.conf：**
```xml
<source>
  @type tail
  @id main-input
  path /var/log/nginx/error.log
  pos_file /var/log/td-agent/nginx-error.pos
  tag nginx.error
  <parse>
    @type nginx_error
  </parse>
</source>

<match nginx.error>
  @type forward
  <server>
    host 10.0.1.3
    port 24224
  </server>
</match>
```

### 优点
- 插件生态最丰富（500+ 插件）
- 支持日志路由和过滤
- 支持多种缓冲方式
- 自动处理日志轮转
- 支持加密

### 缺点
- Ruby 实现，资源占用较高
- 配置语法复杂
- 启动较慢

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| td-agent 无法启动 | 插件加载失败 | `td-agent -vv` 详细调试 |
| 日志延迟严重 | 缓冲区满 | 调整 buffer_type 和 buffer_path |

## 4. nc (netcat)

### 原理
```
tail -f 日志文件 | nc 目标IP 端口
```

### 配置方法
**服务器端：**
```bash
tail -f /var/log/nginx/error.log | nc 10.0.1.3 9999
```

**本地端：**
```bash
nc -l -p 9999 | tee nginx-error.log
```

### 优点
- 系统自带，无需安装
- 命令极简
- 调试方便

### 缺点
- 无重连机制
- 无缓冲，日志丢失风险高
- 无 ACK 确认
- ❌ 不适合生产环境

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| Connection refused | 本地未监听 | 先启动本地 `nc -l -p 9999` |
| 断开后不重连 | nc 不支持 | 改用 socat |

## 5. socat（nc 增强版）

### 原理
```
tail -f 日志 | socat - TCP:目标IP:端口,retry,interval=30
```

### 配置方法
**安装：**
```bash
apt install socat
```

**服务器端（自动重连）：**
```bash
tail -f /var/log/nginx/error.log | \
  socat - TCP:10.0.1.3:9999,retry,interval=5,fork
```

**本地端：**
```bash
socat TCP-LISTEN:9999,fork STDOUT | tee nginx-error.log
```

### 优点
- 自动重连
- 支持 SSL/TLS
- 支持进程管理
- 调试模式丰富

### 缺点
- 需要安装
- 无缓冲队列
- 性能比 rsyslog 低

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| Connection refused | 本地未监听 | 先启动本地 `socat TCP-LISTEN:9999 STDOUT` |
| 重连太频繁 | interval 设太短 | 增加 interval 值 |

## 6. rsync + inotify（定时同步）

### 原理
```
inotifywait 监听文件变化 → rsync 增量同步
```

### 配置方法
**安装：**
```bash
apt install inotify-tools rsync
```

**服务器端脚本 /usr/local/bin/sync-logs.sh：**
```bash
#!/bin/bash
inotifywait -m /var/log/nginx/error.log -e modify -e moved_to |
while read -r path action file; do
    rsync -az --quiet "$path$file" 10.0.1.3:/data/nginx-error.log
done
```

### 优点
- 支持增量同步
- 支持加密（rsync over ssh）
- 不依赖特定端口

### 缺点
- 不是真正实时（有延迟）
- 带宽消耗大
- inotify 有系统上限

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| inotify 达到上限 | 监听过多文件 | `echo 65536 > /proc/sys/fs/inotify/max_user_watches` |
| 同步延迟严重 | 文件过大 | 改用实时方案 |

## 7. HTTP API

### 原理
```
应用/脚本 → HTTP POST → 本地 API → 存储
```

### 配置方法
**服务器端脚本：**
```bash
#!/bin/bash
while true; do
    tail -n 10 /var/log/nginx/error.log | \
    curl -X POST -H "Content-Type: text/plain" \
         -d @- http://10.0.1.3:3000/api/ingest
    sleep 5
done
```

**本地端 API（Node.js）：**
```javascript
const express = require('express');
const app = express();

app.post('/api/ingest', (req, res) => {
  let body = '';
  req.on('data', chunk => body += chunk);
  req.on('end', () => {
    parseLog(body);
    res.json({ success: true });
  });
});

app.listen(3000);
```

### 优点
- 穿透性强（HTTP 端口易放行）
- 有 ACK 确认
- 支持 HTTPS 加密
- 便于负载均衡

### 缺点
- 有轮询延迟
- HTTP 头部开销大
- 需要应用改造或写脚本

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| 413 Request Entity Too Large | 请求体过大 | 减少单次发送量 |
| 429 Too Many Requests | 频率限制 | 增加 sleep 时间 |

## 8. WebSocket

### 原理
```
Agent → WebSocket 连接 → 本地服务 → 浏览器
```

### 配置方法
**服务器端 Agent（Node.js）：**
```javascript
const WebSocket = require('ws');
const tail = require('tail');
const fs = require('fs');

const ws = new WebSocket('ws://10.0.1.3:3000/logs');
const log = new tail.Tail('/var/log/nginx/error.log');

ws.on('open', () => {
  log.on('line', (line) => {
    ws.send(JSON.stringify({
      type: 'nginx-error',
      server: '101.133.135.3',
      timestamp: new Date().toISOString(),
      data: line
    }));
  });
});

ws.on('error', () => {
  setTimeout(() => process.exit(1), 5000);
});
```

### 优点
- 真正实时（毫秒级）
- 双向通信
- 支持 TLS 加密
- 浏览器原生支持

### 缺点
- 需要专门 Agent
- 连接数有限
- 防火墙穿透较弱

### 故障排查

| 故障现象 | 原因 | 解决方案 |
|---------|------|---------|
| WebSocket 403 | Nginx 不支持 | 添加 Upgrade 配置 |
| 连接数爆炸 | 未限制连接数 | 使用连接池和限流 |

## 选择建议

| 场景 | 推荐方案 |
|------|---------|
| 追求简单快速 | socat |
| 生产环境高可靠 | Filebeat |
| 需要复杂处理 | Fluentd |
| 追求原生系统集成 | rsyslog |
| 临时调试 | nc |
| 低频同步 | rsync + cron |

### 方案选择流程图

```
你的需求是什么？
    │
    ├── 临时测试/调试
    │       └── nc（最简单）
    │
    ├── 生产环境，需要可靠
    │       ├── 追求零丢失
    │       │       └── Filebeat（磁盘队列）
    │       │
    │       └── 需要过滤/路由
    │               └── Fluentd（插件丰富）
    │
    ├── 不想装额外软件
    │       ├── 服务器系统自带
    │       │       └── rsyslog
    │       │
    │       └── 通用工具
    │               └── socat
    │
    └── 跨防火墙/外网访问
            └── HTTP API / WebSocket
```

## 总结

| 方案 | 推荐度 | 理由 |
|------|--------|------|
| **Filebeat** | ⭐⭐⭐⭐⭐ | 有缓冲、重连、过滤，最可靠，资源占用低 |
| **Fluentd** | ⭐⭐⭐⭐ | 插件丰富，适合复杂场景，但资源占用高 |
| **socat** | ⭐⭐⭐ | 简单够用，有重连，但无缓冲 |
| **rsyslog** | ⭐⭐⭐ | 原生集成，但配置复杂 |
| **nc** | ⭐ | 仅用于临时调试 |
| **rsync** | ⭐⭐ | 非实时，不推荐 |
| **HTTP API** | ⭐⭐⭐ | 穿透性好，但有延迟 |
| **WebSocket** | ⭐⭐⭐ | 实时，但需要专门 Agent |
